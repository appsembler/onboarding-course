As a code reviewer:

  1. I will give prompt, thoughtful, thorough, constructive code reviews.
  2. (When reviewing OpenCraft work): I will expect that the PR author has done everything outlined in the PR expectations part of the Pull request process section (you'll get to it soon) - if not, I will ask them to fix that before I start the review.
  3. I will always read through the code diff, considering things like:
    • Is the code easily understandable by humans?
    • Is the code of high quality?
    • Are all relevant coding standards followed?
    • Are all UX / a11y / i18n considerations addressed?
    • Is this introducing tech debt?
    • Is the new code well covered by tests?
  4. I will always test the code manually, either locally or on a sandbox, unless there is no reasonable way to do so.
  5. I will always check if any updates to the software's documentation are required, and either ask the author to update the docs, or ensure the relevant documentation team is pinged.
  6. If there is any part of the code that I am not confident in reviewing, I will indicate that in my comments.
  7. If there is any part of the code or PR that I particularly like, I will say so - we want to reinforce good practice as well as flagging issues.
  8. I will set up the following template as a "Saved Reply" in GitHub and use it for the "stamp of approval" on all PRs I review: 
:+1:

- [ ] I tested this: (describe what you tested)
- [ ] I read through the code
- [ ] I checked for accessibility issues
- [ ] Includes documentation
